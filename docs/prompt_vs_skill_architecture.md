# Prompt 與 Skill 架構說明

## 核心差異

### Prompt 型
`Prompt` 是給模型的任務指示文字，用來控制：

- 回覆語氣
- 內容結構
- 輸出格式
- 文案重點

它適合處理「最後要說什麼」。

不適合處理：

- 多步驟流程控制
- 狀態管理
- 資料驗證
- 檔案歸檔邏輯
- 欄位轉換與規則判斷

### Skill / 流程型
`Skill` 在這套系統裡更接近「固定工作流程」或「可重複的能力模組」。

它適合處理：

- 先做什麼、再做什麼
- 要讀哪些資料
- 要怎麼判斷
- 狀態如何流轉
- 何時呼叫 AI
- 何時寫檔、寄信、歸檔

它解決的是「這件事要怎麼做完」。

### 混合型
很多功能不是純 Prompt，也不是純規則，而是：

1. 先用流程模組整理資料與判斷策略
2. 再用 Prompt 生成最終文字

這是目前這套系統最適合的形式。

## 判斷原則

### 適合做成 Prompt 型的情況

- 功能本質是產生文字內容
- 不需要複雜狀態管理
- 主要需求是調整語氣與文案風格
- 同一個流程下，只想換不同寫法

### 適合做成 Skill / 流程型的情況

- 功能涉及多個步驟
- 要讀寫多個資料來源
- 要依條件做規則判斷
- 需要等待下一步輸入
- 需要維護 session / state
- 需要寫入資料、更新欄位、寄信、歸檔

### 適合做成混合型的情況

- 流程固定，但輸出文字要有彈性
- 需要先做策略判斷，再讓 AI 生成內容
- 需要同時兼顧穩定性與可調整性

## 目前系統分類

### Prompt 型

- 課程文宣優化
- 潛在家人邀約文宣
- 跟進夥伴邀約文宣
- 培訓摘要五部分文案
- 激勵訊息語氣與內容
- 營養評估報告文字表達

這些功能的重點是「文字怎麼寫比較好」。

### Skill / 流程型

- 待歸檔兩階段流程
- 行事曆圖片整理
- 語音新增跟進夥伴
- 每日報告寄送
- 夥伴狀態管理
- LINE `5168` 選單與後續編號流程
- Web 表單送出與 API 路由

這些功能的重點是「流程與狀態是否正確」。

### 混合型

- 跟進建議
- 飲食評估
- 課程邀約三段式
- 培訓整理
- 激勵夥伴

這些功能的共同點是：

- 前半段需要流程、資料與規則
- 後半段需要 AI 幫忙寫成可讀內容

## 實務建議

### 什麼應該放進 Prompt

- 可以常常調整的文案
- 不影響資料結構的內容
- 希望由使用者自行微調的描述文字

### 什麼不應該只靠 Prompt

- 狀態定義
- 資料欄位更新規則
- 下拉選單內容
- 跟進流程順序
- 檔案歸檔規則
- 行事曆新增/修改/刪除
- 寄件收件名單與報告區塊結構

### 新功能設計時的判斷問題

新增功能時，可先問三個問題：

1. 這功能主要是在「寫內容」還是在「跑流程」？
2. 如果 AI 暫時失敗，功能是否仍應該能運作？
3. 使用者未來想改的是「文案」還是「規則」？

如果答案偏向：

- 文案：優先做成 Prompt 型
- 規則：優先做成 Skill / 流程型
- 兩者都有：做成混合型

## 對目前系統的結論

目前這套系統最適合採用：

- 流程模組負責狀態、步驟、資料處理
- 提示詞負責最後輸出的文字品質

也就是：

- 不要把所有邏輯都塞進 Prompt
- 也不要把所有文字都硬寫死在程式裡

最穩定的方式是：

1. 用流程模組固定住資料與步驟
2. 用 Prompt 管理可調整的 AI 文案

這樣才能同時保有：

- 穩定性
- 可維護性
- 可調整性
- 擴充性
